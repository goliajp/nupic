# 项目宪法 — 来源与解读

本文件记录 lab29-nupic 立项时用户提出的 6 条硬约束，按原话保留 + 我的解读。

> 来源：2026-06-09 与用户的设计对话。最初语境是 goliajp/admin 已有的 Squoosh-based 图片压缩（jsquash: mozjpeg + oxipng + pngquant），用户问"能否自研一个 pure Rust 的、整体性能上超过 squoosh / pngquant"。在我给出"PNG/JPEG 路径上 pngquant/oxipng 已经是 Rust，重写没意义；mozjpeg 几十年感知调优极难超越"的初版评估后，用户立刻给出了下面 5 条约束（外加项目级别的"业务无关"），把我从"业务 ROI 评估"的思维框架里强行拉回到"数学和物理边界"。

---

## 1. 没有 ROI / win-risk，只有数学和物理边界

**用户原话**：
> 我们完全没有 roi / win/risk 只有数学和物理边界

**解读**：决策依据不是"这个改进值不值"，而是"信息论极限是多少 / 感知模型极限是多少 / 硬件极限是多少"。任何评估都要回到这三类边界之一。

工程含义：
- 不写"性价比不高所以放弃"这种话
- 评估一个改进时，先问"理论上界是多少"，再问"现在离上界差多远"
- 如果一个方向数学上界明确，工程上可达，就做；不看商业回报

## 2. 0 deps，所有路径自研

**用户原话**：
> 我们完全 0 deps 所有路径都自研，完全是能站在一切 rust / c / c++ 优秀的开源项目思想上学习和优化的

**解读**：
- **代码层面**：不依赖外部 codec / 算法库（不引 `imagequant`、`oxipng`、`zopfli-rs`、`ravif`、`zstd-rs`、`flate2` 等任何同领域 crate）。
- **思想层面**：完全可以读 zopfli 论文、imagequant 源码、jxl 参考实现、libdeflate 的 SIMD 套路，学完自己写。
- **工具层面**：编译器、std、SIMD intrinsic 这些当然用。基础数学库（`num`、`rand` 等）按需，但优先自己写紧凑实现。
- **隐含**：因为我们要超越这些开源项目，所以拿它们当依赖在逻辑上自相矛盾。

## 3. 跨平台兼容性是硬约束

**用户原话**：
> 我们可以输出新的 web 兼容的格式，但最好 ios/android 也能兼容，且 win/mac/linux 上能访问直接打开

**解读**：
- 不允许做"只有我们自己能打开"的私有格式。
- 输出格式必须满足：
  - 浏览器原生支持（不依赖我们的 JS 解码器）
  - iOS / Android 系统图库可见可分享
  - Win / Mac / Linux 资源管理器双击能用默认看图工具打开
- 这把我们的"新格式"备选锁定到极少数已被标准化的容器（AVIF / WebP / JPEG XL），不允许发明新文件后缀。
- **PNG 作为输出格式永远满足这条约束**，所以第一条 pipeline 不受这条约束限制。

## 4. 业务无关

**用户原话**：
> 忘记 goliajp 业务场景

**解读**：lab29-nupic 是独立项目。不为 goliajp 的 document 上传 / 图片预览管线做任何特化优化。目标是 codec 工艺本身的顶点，不是业务可用性。

## 5. 一条 pipeline 来做，PNG 优先

**用户原话**：
> 我们肯定是一个个 pipeline 来做，优先还是 png

**解读**：
- **优先级 1：PNG 编码**（含 indexed/truecolor/grayscale, 8/16 bit, alpha, dither）
- 后续 pipeline 待 PNG 完成后再规划。可能候选：AVIF lossless、JPEG XL lossless、自研格式（需先满足 #3）、有损 JPEG/WebP/AVIF。

## 6. 不接受"几十年传统不可超越"

**用户原话**：
> 我们今天是 2026/6 了，不可以再用几十年无法超越去思考，我们站在别的肩膀上进步，这是发展的正确方向

**解读**：
- 这条直接反驳了我初版评估里"mozjpeg 几十年感知调优极难超越"的论证。
- 正确思维方式：mozjpeg 1990s 的算法 + 1976 年的 CIELab + 1976 年的 Floyd-Steinberg，**每一项都被后 30-50 年的研究超越过**（OKLab 2020、SSIMULACRA2 2023、blue-noise dither 1993+、可微分量化 2023+）。我们的工作是把这些"已经出现但还没进 SOTA codec"的研究成果整合进新一代实现。
- "站在肩膀上"不是说"我们再做一遍 mozjpeg"，而是说"我们把 mozjpeg 之后 30 年的研究成果落地"。
